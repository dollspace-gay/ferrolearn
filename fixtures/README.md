# ferrolearn Test Fixtures

This directory contains golden test fixtures generated by scikit-learn for use as
oracle data in ferrolearn's conformance / parity tests.

## Purpose

These fixtures implement the "oracle test" pattern: run the same computation in
scikit-learn (the reference implementation) and in ferrolearn, then compare the
outputs within an algorithm-class-appropriate numerical tolerance. This ensures
algorithmic correctness independent of the specific solver or numerical path
chosen in Rust.

## Fixture Format (schema v2)

Every fixture is a JSON file with the following top-level structure. v1 fixtures
(without the optional `sklearn_pin`, `tolerance`, or `divergence_id` fields)
continue to load correctly — defaults are applied by the oracle.

```json
{
  "description": "Human-readable description of what this fixture tests",
  "sklearn_version": "1.7.2",
  "sklearn_pin": "scikit-learn==1.7.2",
  "random_state": 42,
  "input":  { "X": [[...]], "y": [...] },
  "params": { "fit_intercept": true },
  "expected": { "coefficients": [...], "intercept": 0.123, "predictions": [...] },
  "tolerance":     { "rel": 1e-6, "abs": 1e-9 },
  "divergence_id": null
}
```

| Field            | Type           | Description                                                                     |
|------------------|----------------|---------------------------------------------------------------------------------|
| description      | string         | What the fixture tests                                                          |
| sklearn_version  | string         | sklearn `__version__` recorded at generation time                               |
| sklearn_pin      | string (opt.)  | Pip pin string — added in v2 to surface version drift in PR review              |
| random_state     | int or null    | RNG seed used (null = deterministic without RNG)                                |
| input            | object         | Input data (X, y, etc.) as nested lists                                         |
| params           | object         | Model / transformer hyper-parameters                                            |
| expected         | object         | Expected outputs from the fitted model                                          |
| tolerance        | object (opt.)  | Per-fixture override `{ rel, abs }` — falls back to algorithm-class default     |
| divergence_id    | string (opt.)  | Reference to an entry in `<crate>/tests/conformance/_divergences.toml`          |

Arrays are stored row-major as nested JSON arrays. Special float values are
encoded as JSON strings: `"NaN"`, `"Infinity"`, `"-Infinity"`.

### `divergence_id` semantics

A fixture marked with a `divergence_id` documents a known and **intentional**
divergence from sklearn. The conformance test for such a fixture is
informational only and **must not** be filed as a bug.

A divergence is justified when sklearn is mathematically incorrect (numerical
analysis citation required) or when ferrolearn applies a documented improvement
(commit + reasoning required). See each crate's
`tests/conformance/_divergences.toml` for the registry of accepted divergences.

## Using Fixtures via `ferrolearn-test-oracle`

The oracle crate provides shared loaders and tolerance constants. Per-crate
conformance tests look like:

```rust
use ferrolearn_test_oracle::{
    load_fixture, json_to_array2, json_to_array1,
    assert_close, assert_close_slice,
    TOL_LINEAR_FIT_REL, TOL_LINEAR_FIT_ABS,
};

#[test]
fn linear_regression_matches_sklearn() {
    let fx = load_fixture("linear_regression");
    let x = json_to_array2(&fx.input["X"]);
    let y = json_to_array1(&fx.input["y"]);
    let (rel, abs) = fx.tolerance(TOL_LINEAR_FIT_REL, TOL_LINEAR_FIT_ABS);

    let fitted = ferrolearn_linear::LinearRegression::<f64>::new()
        .fit(&x, &y).unwrap();
    let expected_coefs = json_to_array1(&fx.expected["coefficients"]);
    assert_close_slice(
        fitted.coefficients().as_slice().unwrap(),
        expected_coefs.as_slice().unwrap(),
        rel, abs, "coefficients",
    );
}
```

## Algorithm-Class Tolerance Defaults

Defaults are defined as constants in `ferrolearn-test-oracle`:

| Class                       | REL    | ABS     |
|-----------------------------|--------|---------|
| Direct linear (OLS, Ridge)  | 1e-9   | 1e-12   |
| Iterative linear (Lasso)    | 1e-5   | 1e-8    |
| Logistic / gradient-based   | 1e-4   | 1e-7    |
| Tree predictions            | 1e-9   | 1e-12   |
| Tree ensembles              | 1e-6   | 1e-9    |
| Neighbors                   | 1e-9   | 1e-12   |
| Naive Bayes                 | 1e-9   | 1e-12   |
| Cluster centers             | 1e-4   | 1e-6    |
| Cluster ARI floor           | (≥ 0.95) |       |
| Decomposition               | 1e-6   | 1e-9    |
| Preprocessing               | 1e-10  | 1e-13   |
| Metrics                     | 1e-9   | 1e-12   |
| Kernel methods              | 1e-5   | 1e-8    |
| Covariance                  | 1e-7   | 1e-10   |
| Neural                      | 1e-2   | 1e-4    |
| Model selection             | 1e-9   | 1e-12   |
| Numerical                   | 1e-9   | 1e-12   |

Per-fixture overrides take precedence. Tightening a tolerance for a specific
fixture is a strong claim that ferrolearn matches sklearn to that precision —
prefer tight defaults and loosen only with justification.

## Regenerating Fixtures

```bash
pip install scikit-learn numpy --quiet
python scripts/generate_sklearn_fixtures.py
```

Re-run when:
- scikit-learn is upgraded and you want to track the new reference values.
- New fixtures are added.
- A bug in the generation script is fixed.

Commit the updated script and JSON files together so the fixtures are
reproducible from the script.
